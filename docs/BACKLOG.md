# Tine — Backlog & Roadmap

The public front-door for not-yet-done Tine work: what's next, what's deferred, and what's
explicitly out of scope. (Detailed internal engineering/data-safety debt is tracked separately
and privately; **lsdoc**, the parser, has its own backlog in its own repo.)

**Categories are deliberate and distinct:**
- **In flight** — being built right now.
- **P1 / P2** — will do; P1 next, P2 when P1 clears.
- **Deferred** — genuinely might do later, but no slot yet. *Deferred is NOT WONTFIX.*
- **WONTFIX** — decided we are not doing it, with a reason.

Kept current in the same chunk of work: add items when they surface, remove them when done,
move them between categories in place.

## P2 — backlog (real, lower urgency)

| Item | Notes |

| **Rendered-copy fidelity — still open (math + residual gaps)** | The Jul 3 2026 pass (`f1c5721`) made *Copy / export → Rendered* resolve block refs + user macros. **Still open:** (1) **Math renders as TeX, not typeset/text** — `$…$`/`$$…$$` copy the TeX source. A faithful plain-text form is genuinely lossy (a typeset fraction/integral has none), so this is a per-construct call: keep TeX for display math, but maybe a Unicode approximation or KaTeX text-content for trivial inline math (`E=mc²`). Martin wants this tracked as open, not closed. (2) **Best-effort ref cache** — a `((uuid))` only resolves if it was already rendered on screen in this graph epoch (`resolvedBlockRefSync`); exporting an off-screen subtree, or a ref never rendered, falls back to the bare uuid. Making the export path *await* the async resolve (instead of a sync cache read) would close it. (3) **Built-in/provider macros stay literal** in copy (`{{video}}`/`{{embed}}`/`{{query}}` …) since they're interactive widgets on screen; some have a sensible text form (`{{video url}}` → the url) if wanted. (4) A block ref copies only the referenced block's **first visible line** (matches on-screen `BlockRefView`); resolving a multi-line/block-level target more fully is open. |
| **Sync-conflict file detection + structural merge UI** | Syncthing leaves `*.sync-conflict-*` copies rotting silently in the graph (Martin's real graph has a bunch). Bounded feature: detect them (file-watch + graph scan), surface them in the UI (badge/list, don't index them as pages), and offer a **block-level structural diff/merge** against the winning copy — lsdoc gives block-identity + structure, so this is a per-block 2-way diff with per-hunk keep-mine/keep-theirs triage, not a text-blob diff. Resolve = write merged file, delete the conflict copy (trash, not unlink). Data-safety critical: never auto-merge silently; every resolution is user-confirmed. This attacks the *symptom* Martin actually hits, for days of work instead of the quarter-scale sync engine (see the op-log item under Deferred). Origin: Jul 4 2026 outl study, `subagent-tasks/notes/outl-study.md` §7. |

| **Query-engine indices + finer live-query refresh** (the sheets/supertags pre-req) | **Not needed today — verified Jul 4 2026**, recorded so the design is ready when it is. Current cadence: live `{{query}}` panels re-run at most ~1 Hz while typing (400 ms save debounce `persistence.ts:190` → `cache_gen` bump on save `model.rs:1488` → 700 ms `dataRev` debounce `persistence.ts:141` → panels keyed on `dataRev` re-fetch, `Macro.tsx:142`); each re-run is a native scan over `OnceLock`-cached block facets (`doc.rs:150`, O(1) field checks) ≈ ms per query at current graph size; and scoped eviction #52 (`model.rs:1504`) keeps unaffected memoized results — the keep/evict test is **two-sided** (evict if the page was in the result OR now matches the predicate, re-running the query's real `Pred::eval` against the post-edit doc, `query.rs:292`), fuzz-guarded by `tests/derived_cache_fuzz.rs`. **Becomes real when** query count × graph size × expressiveness grows: supertag/sheets dashboards with many live views, sort/group/join semantics, or much larger graphs. **Design shape:** (1) incremental facet indices (marker / priority / tag / page-ref → block ids) maintained in `cache_upsert` alongside the page cache, so query eval becomes index intersection instead of the `with_pages` full walk (`query.rs:221`); (2) per-query invalidation subscriptions replacing the global `dataRev`-keyed refetch, so an edit only re-renders the queries it actually evicted; (3) diff results before re-render — **DOM churn repainting big result lists ~1 Hz in WebKitGTK is the likely first *felt* cost, not the Rust scan**. **Alternative/complement (Martin's observation, Jul 4 2026): incremental result patching** — the eviction test already evaluates the edited page, so an evicted result could in principle be patched (`old − page's old group + page's new matches`, O(page) per edit) instead of fully recomputed (O(graph), `query.rs:221`). Deliberately NOT done today: global operators break local patching (`sample n` truncation `query.rs:270` — blocks cut by the limit are unknowable without a rescan; same for future sort/limit/group), and the current invariant "a cached result is always a full-recompute output" is what keeps the fuzz oracle trivial and keep/evict drift-free. If sheets adds sort/group/limit, prefer indices (fast recompute) over patching (correctness surface); patch only pure per-block-predicate queries if at all. Risks: index consistency with `cache_gen` + aliases + day-rollover (`today` in memo keys); extend the derived-cache differential fuzz oracle to cover the indices. **Watch signal:** Martin's sidebar TODO-query dashboard (workflow he dropped in OG, resumed Jul 2026) feeling sluggish while typing — first diagnose *which* lags: typing latency (save path; shouldn't happen) vs sidebar repaint stutter (DOM; cap result-list size, collapse coarse queries) vs stale results (harmless debounce). |
| **Block DOM windowing on very large pages** (measurement-gated) | Current state (ADR 0008): lazy render-once-**keep** — block bodies mount via IntersectionObserver-gated `AstBody` (~11/2000 render at load) but **never unmount**, so a long scroll session on a big page accumulates DOM; historically WebKitGTK layout/paint — not parsing — is what a big page *feels* like (per-block parse is µs + LRU-cached; see the scroll/render notes and the #12 jitter lesson: measure in the **real** WebKitGTK app, headless SW rendering does not reproduce compositor behavior). **Measure first:** a long-scroll probe on a synthetic huge page tracking DOM node count / memory / frame time, plus `npm run bench` (`scrollBig`). **Build only if measurement shows it:** true windowing = unmount block bodies scrolled out of the viewport, keeping height placeholders (preserve scroll geometry + anchor targets), or extend the `content-visibility` approach (mind the clipping gotcha from the earlier feed fix). Risks: the just-stabilized caret/focus cluster (ADR 0013 / editorController — never unmount the focused/edited block), block-ref + backlink anchor jumps into unmounted regions, find-in-page, e2e caret-harness assumptions. **Trigger:** a big page is sluggish AND the profile blames layout/paint rather than parse. **May also fix the 4K/1.5× fractional-scaling slowness** (row below): Martin (Jul 4 2026) reports *scrolling* is the worst symptom there, and windowing paints far fewer pixels per frame — so this is the leading candidate lever for that too, but still measurement-gated on Martin's box (headless uni server can't reproduce 4K/GPU). |
| **Performance: remaining levers** | The July 2026 whole-codebase audit (perf × data-safety × architecture, two independent auditors per dimension) closed the found eager/superlinear paths (graph-open foreground scans, bulk icon lookup, batch block-ref resolution, advanced-query memoization). Remaining: block **DOM windowing** (own row above) and flipping the CI bench job from advisory to gating once shared-runner noise is calibrated. | **UI sluggish on a 4K display at fractional scaling** (Martin, Jul 4 2026) | Fast on a native 1920×1200 laptop panel; slow on a 4K screen at **KDE Plasma 1.5× fractional scaling**. **Env workarounds tried and FAILED (Jul 4):** `GDK_SCALE=2 GDK_DPI_SCALE=0.5` → UI too small (effective 1.0×) *and still slow*; `TINE_GPU=0` (software paint) → *also too slow*. That two-way negative is diagnostic: integer 2× backing is slow too (so it's not specifically the fractional-downscale path), and software is slow too (so it's not a GPU/DMABUF-downscale pathology) → the cost is **raw device-pixel paint/composite throughput in WebKitGTK**, proportional to pixel area (1.5× on 4K ≈ 3–4× the 1920×1200 pixel count). No env flag fixes this; it's a paint-budget problem to shrink in *our* code. **Still worth an A/B (confirms + possible daily workaround):** `GDK_SCALE=1 GDK_DPI_SCALE=1.5` — draw at 1× device pixels (cheap), bump only font size to 1.5× (readable), at the cost of small-ish icons/padding; if *this* is fast, it nails the diagnosis. **Real levers (in our control), highest-value first:** (1) a **high-DPI "reduce visual effects" mode** — kill backdrop-filter/blur (modals, overlays), heavy box-shadows, and drop the per-`.query-block` `translateZ(0)` compositing layers ([[tine-rendering-perf]] pt 4; each is a GPU texture that costs ~2.25× RAM/fill at 1.5×) — auto-enable above some device-scale or expose a toggle; (2) land block **DOM windowing** (row above) — fewer painted nodes is the biggest win at 4K; (3) investigate rendering the webview *content* at a capped internal scale (<window scale) and letting the compositor upscale — trades crispness for fill-rate. **Need Martin's box to profile** (uni server is headless, no GPU); the first question is *what* is slow — scroll, typing, or steady-state — since it points at (2)/(1)/(1) respectively. |

Feeds directly into the existing `--ls-*` alias-shim item below. |
| **Slash-menu autocomplete UX** ([#15](https://github.com/martinkoutecky/tine/issues/15)) | Two OG-parity/discoverability gaps in the `/` autocomplete: (1) a **visible scroll affordance** — WebKitGTK hides scrollbars, so overflow items below the 280px fold look absent (affects every menu: slash, block-ref, page); (2) an **OG-style `/template` picker/submenu** — reporter expects Logseq's dedicated template chooser. Backend discovery + listing are verified working; this is the UX layer. Blocked on a repro for the "sees only one template" part. |
| **Raw-HTML rendering via a sanitizer (`<img>` etc.)** ([#16](https://github.com/martinkoutecky/tine/issues/16)) | Markdown should support inline/block raw HTML (OG does), but Tine deliberately renders only a sandboxed-https `<iframe>` today and shows all other raw HTML as text (`renderRawHtml`/`InlineHtmlInline` in `inline.tsx`; lsdoc already *parses* it as `inline_html`/`raw_html`, so parsing is NOT the blocker). **The blocker is safety, and it applies even to a local app:** synced/imported/pasted notes aren't self-authored, and in Tauri an injected `onerror=`/`<script>` can call Tine's IPC to read/write the whole graph or exfiltrate via `fetch`; a remote `<img src>` is a tracking pixel; and the static-HTML **export re-publishes** raw HTML as served content. So the real work is a **sanitizer** (vendor DOMPurify): allowlist tags + per-tag attributes, restrict `src`/`href` schemes to `https:` + Tine's asset protocol, strip event handlers — and it MUST cover the export path too. Bounded slice: enable `<img>` first (`alt`/`width`/`height`/`title`, safe `src` only). **`file://` caveat (surfaces every time #16 comes up):** it's not a WebKitGTK quirk — Tauri serves the app from a locked-down origin (custom protocol + CSP), inherited by **WebView2 on Windows too**, so a bare `<img src="F:\…">`/`file://` won't load; OG dodges this only because it's Electron (loads from `file://`, relaxed `webSecurity`). Tine already shows local images that live **inside the graph** (routed through its asset protocol), so after `<img>` lands: **https images work; arbitrary absolute local paths (like the reporter's `F:\…Joplin…`) still won't** unless the file is moved into the graph or we add an explicit "allow notes to load any local file" opt-in (a real permission decision, not a silent default). Net: the sanitizer is worth doing (md-should-support-html); the specific reporter's Joplin-resource path is only partially helpable. |

## WONTFIX — not doing (with reason)

### By-design non-goals
| Item | Reason |
|---|---|
| **Whiteboards** | A separate application domain; Tine is a fast local-first outliner. |
| **Flashcards / SRS** | Needs a dedicated spaced-repetition review engine; out of scope for an outliner. |
| **Full plugin system (`@logseq/libs`)** | Months of work (a datascript engine + Logseq's render model); Tine coexists with Logseq instead. |
| **Built-in git** | Delegate to your own sync tool (Syncthing, etc.). |

### Subsystem / feature calls
| Item | Reason |
|---|---|
| **`function` macro** | Needs a Clojure (SCI) evaluator plus query-result plumbing. Large, rarely used. |
| **`cloze` / `cards` macros** | Only meaningful inside the SRS review loop → same reason as flashcards. |
| **Zotero connector macros** | Niche; needs Zotero data-dir + item-metadata resolution Tine doesn't have. |
| **CEF / Chromium engine swap (desktop)** | Evaluated and ruled out — too heavy (a multi-process Chromium bundle per OS + a JS-compat shim). Staying on WebKitGTK; Verso is the live long-term bet (Deferred). |
| **YouTube-timestamp seeking** | OG-faithful seeking needs the YouTube player API; not worth it. (A plain clickable timestamp is the most that would be considered.) |
| **Overlay/backdrop "live highlight" editor mirror** | The transparent-textarea-over-rendered-mirror trick can only honestly deliver metric-safe color changes — no chips, no marker hiding, no bold (glyph metrics must stay byte-identical to the textarea) — in exchange for a permanent font-metric-sync liability on WebKitGTK. The CodeMirror 6 route (Deferred) strictly dominates it. |
